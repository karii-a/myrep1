Introduction to Cloud Computing

Lec1_CSVC_Introduction_Cloud_Computing_VE.pptx

What is Cloud Computing?

Cloud Computing is a general term for a new class of network-based computing that takes place over the Internet. It is a collection of integrated and networked hardware, software, and Internet infrastructure (a platform) that hides the complexity of the underlying infrastructure from users via simple graphical interfaces or APIs. The key concept is utility computing — pay only for what you use, available anywhere and anytime.

Simple Definition: A shared pool of configurable computing resources (networks, servers, storage, applications, services) that can be rapidly provisioned and released with minimal management effort or service provider interaction.

Cloud Computing Characteristics

Essential Characteristics

On-Demand Self-Service

Users can provision resources automatically without requiring human interaction with each service provider.

Broad Network Access

Capabilities available over the network, accessible by standard mechanisms via thin or thick clients.

Resource Pooling

Provider's resources pooled to serve multiple consumers using a multi-tenant model, dynamically assigned.

Rapid Elasticity

Capabilities can be elastically provisioned and released to scale rapidly outward and inward with demand.

Measured Service

Resource usage is monitored, controlled, and reported — providing transparency for provider and consumer.

Common Characteristics

Low Cost Software

Use of open-source or commodity software reduces licensing costs.

Virtualization

Abstracts physical resources into logical pools for efficient sharing.

Massive Scale & Homogeneity

Enormous scale with standardized hardware/software for simplicity.

Resilient Computing

Systems designed to recover from failures automatically.

Geographic Distribution

Resources spread across multiple locations for availability and compliance.

Virtualization

Virtualization is the abstraction of physical hardware into multiple virtual machines (VMs). A Hypervisor (also called a Virtual Machine Monitor / VMM) intercepts and emulates instructions from VMs, allowing multiple guest OSes to run on a single physical host.

How It Works

Hardware → Hypervisor (Xen, VMWare, KVM) → Multiple Guest OSes (Linux, Windows, NetBSD) → Apps on each. Para-virtualization (e.g., Xen) achieves near-raw hardware performance.

Virtual Workspaces

An abstraction of an execution environment dynamically available to authorized clients with defined resource quotas (CPU, memory) and software configurations (OS, services).

Cloud Benefit

Clouds use clusters of PC servers + open-source software + in-house applications. Transparent to users — can be built with branded products, proprietary or open source, hardware or software.

Storage Layer — Amazon S3

What is Amazon S3?

Amazon Simple Storage Service (S3) is an object storage service that stores massive, unlimited amounts of unstructured data. Data files are stored as objects inside user-defined buckets. Each object has a REST-accessible globally unique URL, a key, version ID, value, metadata, and subresources.

Key Specs

Maximum single object size: 5 TB  |  Durability: 99.999999999% (eleven 9s)  |  Availability: 99.99% (S3 Standard)

S3 Benefits

Durability

Data is never lost. S3 Standard provides eleven 9s of durability by storing data redundantly across multiple AZs.

Availability

Access your data when needed. Designed for 99.99% availability.

Scalability

Virtually unlimited capacity. Any single object up to 5 TB.

Security

Fine-grained access control at bucket and object level.

Common S3 Use Cases

Store & Distribute Media

Host video, photo, or music uploads/downloads in a redundant, scalable, highly available infrastructure.

Static Website Hosting

Host a fully static website (HTML, CSS, JS, images) directly from S3 with a public URL.

Data Backup & Archive

Store backup copies and archives cost-effectively. Use lifecycle policies to transition to cheaper tiers automatically.

Data Lakes & Analytics

Central repository for raw data at any scale, feeding analytics pipelines.

Access Control

Three general approaches to configuring S3 access:

Default (Private)

By default, only the bucket owner has access. All public access is blocked.

Bucket Policies

JSON-based policies attached to a bucket to grant or deny access to AWS accounts, users, or the public.

Public Access

Explicitly enabled to make content publicly readable — used for static website hosting.

S3 Storage Classes

Storage ClassBest ForKey Trait
S3 StandardFrequently accessed data11 9s durability, 99.99% availability. Highest cost.
S3 Standard-IALong-lived, infrequently accessedLower storage cost but retrieval fee. 99.9% availability.

S3 Lifecycle Policies

Configure an S3 Lifecycle Policy to automatically move or delete objects based on their age. Example: Standard (day 0) → Standard-IA (day 30) → Glacier (day 60) → Delete (day 365).

S3 Pricing

You Are Charged For

GBs of objects stored per month (per Region, per storage class). Transfers OUT to other Regions or the internet. PUT, COPY, POST, LIST, GET, SELECT requests. Lifecycle transitions.

No Charge For

Data transfers IN from the internet. Transfers between S3 buckets or to services in the same Region. Transfers OUT to Amazon CloudFront. DELETE and CANCEL requests.

Moving Data To/From S3

4

5

Choosing a Region

Compliance & Data Residency

Some countries have laws requiring data to stay within borders. Check relevant data privacy laws and governance obligations before selecting a Region.

Proximity to Users

Small latency differences can noticeably impact customer experience. Choose the Region closest to your primary user base.

Service Availability

Not all AWS services are available in all Regions. Consult the AWS Region Table; services expand regularly.

Cost-Effectiveness

Pricing varies by Region. Factor in data transfer costs when replicating environments across Regions.

Amazon EC2

Amazon EC2 provides resizable compute capacity in the cloud. It provisions virtual machine servers in minutes, scales capacity up or down automatically, and charges only for what you use.

Use EC2 when you need: Complete control of computing resources including OS and processor type, options for cost optimization (On-Demand, Reserved, Spot), or the ability to run any type of workload.

Provisioning an EC2 Instance — Key Parameters

AMI

Amazon Machine Image — defines the OS and pre-installed software for the instance.

Instance Type

Determines the CPU, memory, storage, and networking capacity.

Network / VPC

Which VPC and subnet the instance runs in; whether it has a public IP.

Security Group

Acts as a virtual firewall controlling inbound/outbound traffic.

User Data

Bootstrap script that runs when the instance first starts — install packages, configure software.

Storage

Instance Store (ephemeral) or Amazon EBS (persistent block storage).

Amazon Machine Image (AMI)

An AMI is a template containing a root volume (OS + software), launch permissions, and block device mappings. Multiple identical instances can be launched from a single AMI.

Repeatability

Launch identically configured instances from the same AMI repeatedly with precision.

Recoverability

Create an AMI from a configured instance as a restorable backup. Replace a failed instance by launching from the same AMI.

EC2 Instance Types

Instance types define the hardware configuration. They are grouped into families based on use case:

T Family — Burstable

Low baseline with ability to burst above it. Range: 1 vCPU/1 GB RAM → 8 vCPU/32 GB RAM. Ideal for small or variable workloads.

M Family — General Purpose

Balanced compute, memory, and networking. Range: 2 vCPU/8 GB → 96 vCPU/384 GB. Good for query-intensive apps and high connection counts.

R Family — Memory Optimized

High memory relative to CPU. Range: 2 vCPU/16 GB → 96 vCPU/768 GB. Ideal for CPU-intensive and in-memory databases.

Placement Groups

Placement groups control where instances run within an Availability Zone to influence network performance and fault tolerance.

Cluster
Low Latency, High Throughput
All instances in the same high-bisection-bandwidth network segment. Up to 10 Gbps per-flow TCP/IP. Launch all instances at once for best results. For HPC and big data.
Partition
Fault Isolation at Scale
Spreads across logical partitions, each with its own rack (separate network + power). Reduces correlated hardware failure. For Hadoop, Cassandra, Kafka.
Spread
Max Fault Isolation
Each instance on a distinct physical rack. Can span multiple AZs. For small numbers of critical instances that must not fail simultaneously.
Limit

An instance can only be in one placement group at a time. Instances with dedicated host tenancy cannot be placed in a placement group.

Database Layer — Amazon RDS & DynamoDB

Lec4_CSVC_Database_Layer_VE.pptx

Database Selection Considerations

Scalability

How much throughput is needed? Can the solution scale up later to handle growth?

Storage Requirements

How large will the database be — GB, TB, or petabytes of data?

Object Size & Type

Simple data structures, large objects, or both? Structured vs unstructured?

Durability

Required level of data durability, availability, recoverability. Any regulatory obligations?

Relational vs Non-Relational Databases

Relational (SQL) — RDBMS

Best when: You need strict schema rules, ACID compliance, data quality enforcement, and moderate read/write capacity.

Benefits: Ease of use, data integrity, reduced storage, standard SQL language.

Examples: MySQL, Oracle, Microsoft SQL Server, PostgreSQL.

Non-Relational (NoSQL)

Best when: You need horizontal scaling for massive data volume, data doesn't fit traditional schemas, or read/write rates exceed RDBMS capacity.

Benefits: Flexibility, horizontal scalability, high performance, highly functional APIs.

Examples: MongoDB, Cassandra, Redis.

Amazon RDS — Relational Database Service

Amazon RDS is a fully managed relational database service. AWS handles power, hardware, OS installation, patching, backups, scaling, and high availability — so you can focus on query optimization and application logic.

Supported engines: MySQL, PostgreSQL, MariaDB, Oracle, Microsoft SQL Server, and Amazon Aurora.

RDS High Availability — Multi-AZ

Multi-AZ deploys a synchronous standby replica in a different Availability Zone. Automatic failover occurs (via Amazon Route 53) if the primary instance fails.

Automatic Failover Conditions

Loss of availability in primary AZ, loss of network connectivity to primary, compute unit failure, or storage failure on primary.

Benefits

Enhanced durability, increased availability, zero-downtime failover (automatic), and simplified operations.

RDS Read Replicas — Performance

Read replicas use asynchronous replication to offload read traffic from the primary instance. Up to 5 read replicas per primary. Supported by MySQL, MariaDB, PostgreSQL, Oracle. For strict read-after-write consistency, always read from the primary.

Amazon Aurora

Amazon Aurora is a fully managed, MySQL- and PostgreSQL-compatible relational engine. It provides up to 5× the throughput of MySQL and up to 3× the throughput of PostgreSQL. Replicates data 6 ways across 3 Availability Zones automatically.

Amazon Redshift — Data Warehousing

Amazon Redshift is used for Online Analytics Processing (OLAP). It stores very large, highly structured datasets and can query exabytes of data in Amazon S3 as well. Use Redshift for business intelligence and analytics, not transactional workloads.

Amazon DynamoDB — NoSQL

DynamoDB is a fully managed, serverless, key-value and document NoSQL database. It provides single-digit millisecond latency at any scale and is ideal for event-driven and serverless architectures.

Extreme Horizontal Scaling

Scales to handle millions of requests per second automatically, with no downtime.

Global Tables

Replicates data automatically to multiple AWS Regions for globally distributed, low-latency access.

Serverless

No servers to provision or manage. Fully managed by AWS — you only pay for reads/writes.

Consistency Options

Eventually Consistent (default): All copies usually consistent within 1 second. Strongly Consistent (optional): Reflects all writes that occurred before the read.

Database Security Controls

ControlFor RDSFor DynamoDB
Network IsolationRun in a VPC — service isolation + IP firewall protection.Configure VPC endpoints to prevent traffic traversing open internet.
AuthenticationIAM policies determine who can manage RDS resources.IAM roles to authenticate access; define fine-grained API permissions.
Access ControlSecurity groups control which IPs or EC2 instances can connect. Network access disabled by default.IAM policies at table, item, or attribute level. Principle of least privilege.
Encryption in TransitUse SSL/TLS for connection encryption.All communications use HTTPS by default.

AWS Database Migration Service (AWS DMS)

AWS DMS migrates data to and from most commercial and open-source databases. Supports both homogeneous (same engine) and heterogeneous (different engine) migrations. The AWS Schema Conversion Tool (AWS SCT) converts the schema when changing engines.

Migration Steps

1. Create a target database  →  2. Migrate the schema  →  3. Set up data replication  →  4. Initiate data transfer  →  5. Switch production to new database

If the database is too large to migrate over the internet, use AWS Snowball Edge — multi-terabyte physical transfer without using the internet.

Microservices & Serverless Architecture

Lec10_CSVC_Microservices_Serverless_Archi_VE.pptx — ECS & AWS Lambda

What Are Microservices?

Microservices are applications composed of independent services that communicate over well-defined APIs. Each service handles a specific business function and can be developed, deployed, and scaled independently.

Monolithic Application

All functions (Users, Topics, Messages) are tightly coupled in a single deployment unit. A change to one function requires redeploying the entire application. Hard to scale individual parts.

Microservice Application

Each function (Users service, Topics service, Messages service) is an independent service. Can be updated, scaled, and deployed individually without affecting others.

Characteristics of Microservices

Decentralized

No single controlling authority. Each service owns its own data and decision-making.

Independent

Services can be deployed, updated, and scaled without coordinating with other services.

Black Boxes

Internal implementation is hidden. Services communicate only via well-defined APIs.

Polyglot

Different services can use different programming languages, databases, and frameworks.

Specialized

Each service is focused on doing one thing very well.

You Build It, You Run It

The team that builds a service is responsible for operating it in production.

Containers & Amazon ECS

A container packages your application code, dependencies, configurations, and runtime engine together so it runs reliably in any environment — from a developer's workstation to production.

Amazon ECS (Elastic Container Service) is a fully managed container orchestration service. It manages when containers run, maintains and scales the fleet of instances running containers, and removes infrastructure complexity.

ECS Launch Types

EC2 Launch Type

You manage the underlying EC2 instances (container instances). ECS schedules containers onto these instances. Gives you more control and visibility into the infrastructure.

Fargate Launch Type

Fully serverless — AWS manages the underlying infrastructure entirely. You only define CPU and memory requirements. No EC2 instances to manage. Easiest path.

Decomposing Monoliths with ECS

1
Create Container ImagesCreate a separate Docker image for each microservice and push them to Amazon ECR (Elastic Container Registry).
2
Create Task Definitions & Target GroupsDefine launch type, container image URL, CPU/memory allocation, and ports for each service. Create a target group for each service.
3
Connect Load Balancer to ServicesUse an Application Load Balancer with listener rules: IF path = /api/users* THEN forward to users target group. Each service gets its own route.

Supporting Tools

AWS Cloud Map

Fully managed service discovery. Defines custom names for application resources and maintains updated locations of dynamically changing resources.

AWS App Mesh

Service mesh that captures metrics, logs, and traces from microservices. Controls traffic flows between services for high availability. Exports to CloudWatch and X-Ray.

Serverless Architecture

Serverless means building and running applications without thinking about servers. AWS manages all the infrastructure provisioning, maintenance, and scaling.

No Infrastructure Management

No provisioning, patching, or maintaining servers. AWS handles everything below the application code.

Automatic Scaling

Scales automatically from zero to millions of requests without any configuration.

Pay for Value

Pay only when your code runs — no charge for idle time. Significantly lower total cost of ownership.

Highly Available & Secure

Built-in availability and security across multiple AZs without any extra work.

AWS Lambda

AWS Lambda is a fully managed compute service that runs your code in response to events — changes to S3, DynamoDB, scheduled cron jobs, HTTP requests, etc. Supports Node.js, Python, Java, C#, Go, Ruby, PowerShell, and Runtime API.

Anatomy of a Lambda Function: handler(event, context) — The handler is the function invoked; the event object carries data sent during invocation; the context object provides methods to interact with runtime information (request ID, log group, etc.).

Lambda Configuration & Billing

Memory

You configure memory allocation. Cost per 100 ms of execution duration increases as memory increases.

Timeout

You set the maximum execution duration for your function.

Pricing

Charged based on number of requests + duration (measured in GB-seconds). No charge when idle.

Lambda Layers

Up to 5 layers (250 MB total) to share reusable code/dependencies across functions. Keeps deployment packages small and promotes separation of responsibilities.

Amazon API Gateway

API Gateway is a fully managed service for creating, publishing, and managing APIs at any scale. It acts as the "front door" for applications to access data, business logic, or functionality from backend services (including Lambda). Common architecture: Client → API Gateway → Lambda → Database.

AWS Step Functions — Orchestrating Microservices

Step Functions coordinates microservices using visual workflows (state machines). It automatically triggers and tracks each step, and provides simple error catching and logging.

State Types in AWS Step Functions

State TypePurpose
TaskA single unit of work performed by the state machine (e.g., invoke a Lambda function).

ParallelCreates parallel branches to run simultaneously in the state machine.
MapDynamically iterates over a list of items, running steps for each element.

State machines are defined in JSON using the Amazon States Language (ASL). You specify a StartAt state, then define each States object with its Type, Resource (ARN), and Next state. The workflow is also visualized in the Step Functions console.

Automating Your Architecture

Lec11_CSVC_Automating_Architecture_VE.pptx — AWS CloudFormation & Elastic Beanstalk

Why Automate?

Manual infrastructure processes create serious problems at scale:

No Repeatability at Scale

How do you replicate the same deployment across multiple Regions consistently?

No Version Control

How do you roll back to a prior version of the production environment if something breaks?

No Audit Trail

How do you ensure compliance and track resource-level configuration changes over time?

Inconsistent Data

How do you guarantee identical configurations across multiple EC2 instances?

AWS Well-Architected Framework: "Perform operations as code" and "Make frequent, small, reversible changes" (Operational Excellence). "Manage change in automation" (Reliability). Manual processes cannot meet these guidelines.

AWS CloudFormation — Infrastructure as Code (IaC)

CloudFormation is AWS's built-in IaC tool. It lets you model, create, and manage a collection of AWS resources — called a stack — using a template file. No extra charge; you pay only for the resources created.

1
Define Resources in a TemplateWrite a YAML or JSON template file specifying all the AWS resources to create (EC2 instances, VPCs, S3 buckets, security groups, etc.).
2
Upload the TemplateUpload directly to CloudFormation or point to a template stored in an S3 bucket.
3
Execute Create StackResources are created across multiple AWS services in your account as a running environment.
4
Manage the StackThe stack retains control. You can later update, detect drift, or delete the stack — which also deletes all resources it created.

Infrastructure as Code Benefits

Reusability

One template can deploy identical environments (dev, test, prod) on demand.

Repeatability

Every deployment from the same template is guaranteed to be identical. No human error.

Maintainability

Modify the template, run Update Stack on all stacks to propagate changes simultaneously.

Easy Cleanup

Deleting the stack deletes all resources it created — no orphaned infrastructure.

CloudFormation Template Structure

SectionPurposeRequired?
AWSTemplateFormatVersionSpecifies the template format version.Optional
ParametersValues set at runtime (e.g., environment type, key pair name).Optional
ResourcesDefines what AWS resources to create. Can reference Parameters.✅ Required
OutputsValues returned after stack creation (e.g., instance ID, public IP).Optional
JSON vs YAML: YAML is less verbose (no {}, ""), supports comments, and is the recommended format. JSON is more widely used by APIs. Both are fully supported. Treat templates as source code — store in a version control repository.

CloudFormation Advanced Features

Change Sets

Preview changes to a stack before executing them. Shows you exactly what will be added, modified, or deleted without actually making changes.

Drift Detection

Detects when actual deployed resources differ from the template. If someone manually modifies a resource, drift detection shows which resources are MODIFIED vs IN_SYNC.

Conditions

A single template can deploy different infrastructure for different environments (e.g., multi-AZ for production but single-AZ for development).

Quick Start Templates

Pre-built reference architectures from AWS and AWS Partners. Start from a vetted best-practice template instead of from scratch.

AWS Systems Manager

Systems Manager gives you operational visibility and control over AWS infrastructure. Key capabilities include Run Command (run scripts on fleets of instances without SSH), Automation (define runbooks for operational tasks), and Parameter Store (securely store configuration data and secrets).

AWS OpsWorks

OpsWorks is a configuration management service that automates how servers are configured, deployed, and managed. It provides managed instances of Chef and Puppet — popular automation platforms.

OpsWorks for Chef Automate

Uses Chef cookbooks and recipes to configure servers with reusable, testable code.

OpsWorks for Puppet Enterprise

Uses Puppet modules to manage server state declaratively.

OpsWorks Stacks

Model your application as a stack of layers (load balancing, application server, database). Complements CloudFormation: use CF to build the infrastructure, OpsWorks to deploy the application layer on top.

AWS Elastic Beanstalk

Elastic Beanstalk is an easy, managed service for deploying web applications. You upload your code and Beanstalk automatically handles infrastructure provisioning, load balancing, auto scaling, health monitoring, analysis, debugging, and logging. No additional charge — pay only for the underlying resources used.

Supported Platforms: Java, .NET, PHP, Node.js, Python, Ruby, Go, and Docker. Deploys on Apache, NGINX, Passenger, Puma, and IIS web servers automatically.

Choosing the Right Automation Tool

ToolControl LevelBest For
Do It Yourself (EC2)Maximum controlFull custom configuration; everything manually managed.
AWS CloudFormationHigh (IaC)Repeatable, version-controlled infrastructure deployment. Best for complex architectures.
AWS OpsWorksMediumConfiguration management with Chef/Puppet for ongoing server state management.
AWS Elastic BeanstalkLower (convenience)Simple web app deployments where you want to focus only on code.
Exam Tip — Automation Rule of Thumb

CloudFormation: Use when you need full control over AWS infrastructure provisioning and want IaC. Elastic Beanstalk: Use when you want the simplest path to deploy a standard web app and don't care about the underlying infrastructure. OpsWorks: Use when you have existing Chef/Puppet expertise and need ongoing configuration management.

Networking Layer — Amazon VPC

Lec5_CSVC_Networking_Layer_VE.pptx — Creating a Networking Environment

What is Amazon VPC?

Amazon Virtual Private Cloud (VPC) lets you provision a logically isolated section of the AWS Cloud where you can launch AWS resources in a virtual network that you define. You bring your own IP addresses, subnets, routing rules, network configuration, and security rules — full control over your virtual networking environment.

VPC Scope: A VPC belongs to only one AWS Region but can span all Availability Zones within that Region. A subnet, however, belongs to exactly one Availability Zone. AWS limits 5 VPCs per Region per account (soft limit, can be increased).

CIDR & Subnets

Classless Inter-Domain Routing (CIDR) notation defines IP ranges. A VPC is divided into subnets — each is a partition of the VPC's IP address range allocated to a group of resources.

CIDRTotal IPsNotes
/2816Smallest practical subnet
/24256 (251 usable)AWS reserves 5 IPs per subnet
/204,096Good mid-size subnet
/1665,536Maximum VPC size
Subnet Rules

Subnets are not isolation boundaries — they are address partitions. Subnet CIDR blocks cannot overlap. Each subnet resides entirely within one Availability Zone. AWS reserves 5 IP addresses per subnet (network, VPC router, DNS, future use, broadcast).

VPC Design Best Practices

One Subnet per AZ

Create one subnet per available Availability Zone for each group of hosts that have unique routing requirements.

Divide Evenly

Divide your VPC network range evenly across all available AZs in a Region to maximize fault tolerance.

Reserve Space

Do not allocate all network addresses at once — reserve space for future use and growth.

No Overlap

Ensure your VPC CIDR block does not overlap with your organization's other private network ranges.

Single vs. Multiple VPCs vs. Multiple Accounts

PatternBest Suited ForConsideration
Single VPCSmall single applications, HPC, identity managementLimited use cases only
Multiple VPCsSingle team / single organization, managed service providersEasier to maintain standards with limited teams
Multiple AccountsLarge organizations, multiple IT teams, rapid-growth companiesBetter isolation; harder to manage access standards

Connecting VPCs to the Internet

1
Internet Gateway (IGW)Allows communication between VPC resources and the internet. Horizontally scaled, redundant, and highly available by default. Provides a target in route tables for internet-routable traffic. Attach one to your VPC to make a subnet "public".
2
Route TablesRequired to direct traffic between VPC resources. Each VPC has a main (default) route table. Best practice: use custom route tables for each subnet. A route to 0.0.0.0/0 pointing at an IGW makes traffic internet-routable.
3
Elastic IP AddressesStatic, public IPv4 addresses associated with your AWS account. Can be associated with an instance or elastic network interface. Can be remapped to another instance — useful for redundancy when load balancers are not an option.
4
NAT GatewayEnables instances in a private subnet to initiate outbound traffic to the internet or other AWS services, while preventing the internet from initiating connections to private instances. Sits in a public subnet; private route table points 0.0.0.0/0 to the NAT gateway.
5
Bastion HostA server in the public subnet whose purpose is to provide SSH/RDP access to instances in private subnets from an external network. Must minimize chances of penetration. Acts as a jump server.

Securing the Network — Stateful vs. Stateless Firewalls

Security Groups — Instance Level

Stateful Firewall

Controls inbound and outbound traffic to AWS resources. Acts at the level of the instance or network interface. Return traffic is automatically allowed — you only need to define inbound rules. Default: block all inbound, allow all outbound.

Chaining Security Groups

You can chain SGs across tiers: Web tier allows HTTP/HTTPS from the internet; App tier allows port 8000 only from Web tier SG; DB tier allows port 3306 only from App tier SG. All other ports blocked by default.

Network ACLs — Subnet Level

Stateless Firewall

Acts at the subnet level. Default ACL allows all inbound and outbound traffic. Stateless means explicit rules are required for both inbound AND outbound traffic. Rules are evaluated in number order (lowest first), with a catch-all DENY * at the end.

When to Use ACLs

Recommended only for specific network security requirements. For most use cases, security groups are sufficient. Use NACLs as an additional layer of defense for specific subnet-level restrictions.

How to Create a Public Subnet (Checklist): (1) Attach an Internet Gateway to your VPC. (2) Point the subnet's route table to the IGW for 0.0.0.0/0. (3) Ensure instances have public IP or Elastic IP addresses. (4) Configure security groups and NACLs to allow relevant traffic.
Exam Tip — Security Groups vs. NACLs

Security Groups: stateful, instance-level, only allow rules, return traffic automatic. NACLs: stateless, subnet-level, allow AND deny rules, must explicitly allow return traffic. Structure your infrastructure with multiple layers of defense — NACLs + SGs together.

Securing User Identity — AWS IAM

Lec6_CSVC_Securing_User_Identity_Layer_VE.pptx — Securing User and Application Access

The Root Account & Why to Avoid It

The account root user has a large amount of power and full, unrestricted access to all AWS services. Best practice: (1) Create an IAM admin user immediately. (2) Lock away the root user credentials. (3) Use the IAM user for all subsequent tasks — console, CLI, or SDK.

IAM Core Components

IAM User

An identity defined in your AWS account. Uses long-term credentials (username/password or access keys) to authenticate programmatically or via the AWS Management Console.

IAM Group

A collection of IAM users granted identical authorization. Makes it easier to manage access across multiple users. Best practice: create groups that reflect job functions (e.g., Developers, Testers, Admins).

IAM Policy

A JSON document that defines which resources can be accessed and at what level. Attach to users, groups, or roles. Explicit Deny always overrides any Allow statement.

IAM Role

Provides temporary security credentials. Not uniquely associated with one person — assumable by a person, application, or service. Used for delegation and cross-account access.

IAM Policy Structure

Policies are JSON documents with a Statement array. Each statement contains:

FieldPurposeExample
EffectAllow or Deny the action"Effect": "Allow"
ActionType of access (API operation)"Action": "s3:GetObject"
ResourceThe AWS resource the action applies to (ARN)"Resource": "arn:aws:s3:::my-bucket/*"
ConditionWhen the rule applies (optional)"StringEquals": {"aws:username": "johndoe"}
Permission Evaluation Logic: Is the action explicitly denied? → DENY. Is the action explicitly allowed? → ALLOW. Otherwise → Implicit DENY. An explicit Deny always wins, even if an Allow statement also exists.

Identity-Based vs. Resource-Based Policies

Identity-Based Policies

Attached to a user, group, or role. Three types: AWS Managed (pre-built by AWS), Customer Managed (you create and control), and Inline (embedded directly in a single principal).

Resource-Based Policies

Attached directly to an AWS resource (e.g., an S3 bucket). Always an inline policy. Specifies who (Principal) can access the resource and what they can do.

IAM Groups & Least Privilege

Use IAM groups to grant the same access rights to multiple users. All users in the group inherit the group's permissions. Best practice: follow the principle of least privilege — grant only the permissions required to perform a task.

Tip — Combine Approaches: Add a user to a group for standard job-function access (e.g., Developer group). Optionally attach an additional policy directly to the user for specific exceptions. Users can belong to more than one group — the most restrictive policy then applies.

IAM Roles & Temporary Credentials

1
Define the IAM PolicyWrite a policy document granting the needed permissions (e.g., read/write access to an S3 bucket).
2
Attach the Policy to a RoleCreate an IAM role and attach the policy to it. A role is not tied to a specific person — it's a set of permissions.
3
Assume the RoleAn EC2 instance, Lambda function, or IAM user assumes the role via AWS STS. They receive temporary security credentials (time-limited access key + secret + session token).

AWS STS — Security Token Service

AWS STS is a web service that enables you to request temporary, limited-privilege credentials for IAM users or for federated users. Credentials expire automatically — no long-term access keys to manage.

RBAC vs. ABAC

RBAC — Role-Based Access Control
Traditional Approach
Create distinct IAM roles for each permission combination (e.g., database-admin, read-only-dev). Update permissions by adding access for each new resource — can become time-consuming as the organization grows.
ABAC — Attribute-Based Access Control
Highly Scalable Approach
Permissions rules are based on attributes (tags) — e.g., Team=Developers, Project=Unicorn. Permissions automatically apply based on matching tags. Granular without needing to update policies for every new user or resource.

Identity Federation

Identity federation allows authentication to be completed by a system external to the AWS account (e.g., corporate directory). Provides a way to allow access through existing identities, without creating IAM users.

AWS STS & Custom Broker

A custom identity broker application authenticates the user against a corporate directory, then requests temporary credentials from STS, and passes them to the application.

SAML Federation

User navigates to a portal (IdP). IdP authenticates user and returns a SAML assertion. The assertion is posted to the AWS sign-in endpoint. AWS STS issues temporary credentials → user is redirected to the console.

Amazon Cognito

Fully managed service providing authentication, authorization, and user management for web and mobile apps. Supports federated identities (Amazon, Facebook, Google, SAML) and User Pools (maintain your own directory). Acts as an OIDC-compatible identity broker.

Multiple AWS Accounts & AWS Organizations

Most organizations choose multiple accounts to: isolate business units, separate dev/test/prod environments, isolate auditing data, and handle billing per unit. AWS Organizations solves the management challenge:

Organizational Units (OUs)

Group accounts into a hierarchy of OUs (e.g., Internal IT OU → Engineering OU → Dev OU, Prod OU). Policies attached to a parent OU are inherited by all child accounts.

Service Control Policies (SCPs)

Control which AWS services are accessible to IAM users in member accounts. SCPs cannot be overridden by the local administrator. Example: deny all members from disabling CloudTrail. IAM policies in individual accounts still apply — SCPs set the maximum permissions boundary.

Consolidated Billing

One payment method for all accounts. Volume discounts aggregate across the organization. AWS Organizations also provides automated account creation via API.

AWS CloudTrail

CloudTrail logs and monitors all user activity across your AWS account. It provides a 90-day event history by default (at no cost) covering all actions taken through the Console, CLI, and SDKs. Use it to identify who accessed your account, when, from where, and what action they took — critical for security analysis and compliance.

Exam Tip — IAM Quick Rules

Explicit Deny always wins. Default (no policy) = implicit deny. Best practice = principle of least privilege. Use groups for job functions, not individual user policies. Use roles for services and cross-account access — never long-term credentials on EC2. Federation via Cognito or SAML for external users.

High Availability & Elasticity

Lec7_CSVC_High_Availability_Architecture.pptx — EC2 Auto Scaling, ELB, Route 53, CloudWatch

Elasticity vs. Scalability

An elastic infrastructure can expand and contract as capacity needs change — scaling web servers when traffic spikes, lowering write capacity when traffic drops, handling day-to-day fluctuations. Scaling is the technique used to achieve elasticity.

Horizontal Scaling

Scale out (launch more instances) and scale in (terminate instances). Best paired with a load balancer. Preferred approach for stateless applications — effectively unlimited scale ceiling.

Vertical Scaling

Scale up (increase instance size) or scale down (decrease instance size). Simple but has a hard limit — only so much CPU/RAM one machine can have. Requires a restart in most cases.

Amazon EC2 Auto Scaling

EC2 Auto Scaling automatically launches or terminates instances based on specified conditions. It registers new instances with load balancers and can launch across Availability Zones for fault tolerance.

Scaling Options

Scheduled Scaling

Scale based on date and time. Good for predictable workloads. Example: turn off dev/test instances at night, scale up for a known marketing campaign.

Dynamic Scaling

Responds to changing conditions in real time. Supports target tracking (keep CPU at 50%), step scaling (scale proportionally to alarm breach), or simple scaling (single adjustment). Good for unpredictable workloads.

Predictive Scaling

Uses machine learning to predict demand and proactively provision capacity. Good for predicted demand with complex patterns (e.g., ecommerce during a major sales event).

Dynamic Scaling Policy Types

Policy TypeHow It WorksBest For
Simple ScalingSingle scaling adjustment triggered by a CloudWatch alarmNew workloads, spiky workloads
Step ScalingAdjustment amount depends on size of alarm breachPredictable workloads with known patterns
Target TrackingMaintains a target value for a specific metric (e.g., 50% CPU)Load-balanced and batch processing applications

Auto Scaling Group Configuration

An Auto Scaling group defines three capacity values: Minimum (never go below this), Maximum (never exceed this), and Desired (the current target number of instances, which fluctuates in response to events). You can mix On-Demand and Spot Instances in a single ASG for cost optimization.

Scaling Databases

RDS Vertical Scaling

Push-button scaling — scale DB instances vertically from micro to 24xlarge with minimal downtime. Simple but limited by instance size ceiling.

RDS Read Replicas

Horizontal scaling for read-heavy workloads. Up to 5 read replicas (15 for Aurora). Replication is asynchronous. Available for MySQL, MariaDB, PostgreSQL, Oracle.

Aurora Serverless

Automatically scales capacity, starts up, and shuts down. Pay for Aurora Capacity Units (ACUs) used. Good for intermittent and unpredictable workloads.

Database Sharding

Split data into large chunks (shards) across multiple databases. Example: even-numbered employee IDs in one DB, odd-numbered in another. Improves write performance.

DynamoDB On-Demand

Pay per request — no provisioning needed. Rapidly accommodates need. For spiky, unpredictable workloads.

DynamoDB Auto Scaling

Default for all new tables. Specify upper and lower bounds. Uses Application Auto Scaling + CloudWatch alarms to dynamically adjust provisioned throughput.

DynamoDB Adaptive Capacity: Automatically increases throughput capacity for "hot" partitions that receive more traffic (enabled by default for every table). Note: it does NOT fix bad partition key design — e.g., a status code with only a few values as a partition key is still a bad choice.

Elastic Load Balancing (ELB)

A managed load balancing service that distributes incoming application traffic across multiple EC2 instances, containers, IP addresses, and Lambda functions. Can be external-facing or internal-facing. Each load balancer receives a DNS name and recognizes/responds to unhealthy instances.

Implementing High Availability

Start with two Availability Zones per AWS Region. If resources in one AZ are unreachable, the application should not fail. The ELB automatically routes traffic only to healthy instances in the available AZ.

HA Architecture Pattern: Internet → Internet Gateway → Application Load Balancer (public subnets in 2+ AZs) → EC2 Auto Scaling group (private subnets in 2+ AZs) → NAT Gateways (one per AZ for outbound private traffic) → RDS Multi-AZ standby.

Amazon Route 53

Route 53 is a highly available and scalable cloud DNS service. It translates domain names into IP addresses, connects user requests to infrastructure inside and outside of AWS, and can monitor health of endpoints for failover.

Simple

Single resource behind the domain — no health checks.

Weighted Round Robin

Route different percentages of traffic to different resources (A/B testing, gradual migrations).

Latency-Based

Route to the region with the lowest latency for the user.

Geolocation

Route based on geographic location of the user (country, continent).

Failover

Active-passive failover — route traffic to primary; if health check fails, route to secondary. Key for DR.

Multivalue Answer

Returns multiple IP addresses — can function as a simple load balancer with health checks.

Monitoring with Amazon CloudWatch

CloudWatch collects and tracks metrics, logs, and events for your resources and applications. It visualizes data on dashboards, enables alarms, and can trigger automated responses.

1
MetricsCloudWatch collects metrics (CPU utilization, disk I/O, network traffic) from AWS services and custom applications. Metric data is retained for 15 months.
2
LogsCollect and monitor log files from EC2, Lambda, VPC Flow Logs, Route 53, ELB, and custom applications.
3
AlarmsTrigger actions when a metric breaches a threshold. Examples: send SNS notification, trigger Auto Scaling to add an instance, run a Lambda function.
4
Amazon EventBridgeA serverless event bus that routes real-time data from AWS services, SaaS apps, and custom applications to targets (Lambda, SQS, SNS, Step Functions, etc.) based on rules.
Exam Tip — High Availability Quick Summary

Use multiple AZs for fault tolerance. Use ALB for Layer 7, NLB for Layer 4 ultra performance. Use Auto Scaling for elasticity (scheduled, dynamic, predictive). Use Route 53 Failover routing for multi-region DR. Use CloudWatch for metrics, alarms, and automated responses.

Caching Content

Lec8_CSVC_Caching_Content_VE.pptx — CloudFront, ElastiCache, Session Management

What Is Caching?

A cache is a high-speed data storage layer that stores a subset of data, typically transient in nature, so that future requests for that data are served faster. It trades storage capacity for speed, reducing the need to access the underlying slower storage layer.

What to Cache

Data that requires a slow and expensive query to acquire. Relatively static and frequently accessed data (e.g., user profiles). Information that can be stale for some time (e.g., a publicly traded stock price).

Benefits

Improves application speed. Reduces response latency. Reduces database access time. Reduces cost by offloading read traffic from expensive database instances.

Caching Layers

Caches can be applied at multiple layers: client (browser), CDN/edge (CloudFront), web/application server, and database (ElastiCache, DAX). Each layer reduces load on the next.

Edge Caching — Amazon CloudFront

A Content Delivery Network (CDN) is a globally distributed system of caching servers that caches copies of commonly requested files (static content) and delivers a local copy from a nearby edge location, reducing latency.

Amazon CloudFront is AWS's global CDN. It is optimized for all delivery use cases with a multi-tier cache by default, provides an extra layer of security, and supports WebSockets and HTTP/HTTPS.

1
Specify an Origin ServerThe source of your content: S3 bucket, EC2 instance, Application Load Balancer, MediaPackage channel, MediaStore container, or any custom HTTP server.
2
Configure the DistributionSet access controls, security settings, cookie/query-string forwarding, geo-restrictions, and access logs.
3
CloudFront Assigns a DomainCloudFront assigns a domain name (e.g., d111111abcdef8.cloudfront.net) which can be aliased to your custom domain.
4
Configuration Distributed to Edge LocationsCloudFront sends your distribution's configuration to its global network of edge locations. Requests are routed to the nearest edge location.

How CloudFront Serves Content

When a user requests content, the request is routed to the nearest edge location. If the content is cached (cache hit), it is served directly. If not (cache miss), CloudFront retrieves the content from the origin server, caches it at the edge location, and delivers it to the user.

Content Expiration Strategies

TTL (Time to Live)

Set a fixed expiration period. CloudFront uses an If-Modified-Since header when checking with the origin. Best for content that changes on a known schedule.

Change Object Name

Rename the file (e.g., header-v1.jpg → header-v2.jpg). The new name forces an immediate cache refresh at edge locations. Most efficient method for immediate updates.

Invalidate Object

Manually remove an object from edge caches. Last resort — inefficient and incurs costs. Use only when immediate removal is critical and renaming is not feasible.

CloudFront Use Cases

Video on Demand

Source video → AWS Elemental MediaConvert → multiple output formats in S3 → CloudFront delivers adaptive bitrate stream to users on any device.

Dynamically Generated Content

Even dynamically generated content (e.g., map tiles) can be cached at CloudFront edge locations after the first request, dramatically reducing DB load.

DDoS Protection

CloudFront (combined with AWS WAF and Route 53) absorbs DDoS traffic at the edge, protecting your origin infrastructure. Malicious traffic is filtered before reaching your VPC.

Session Management

Web sessions store user authentication and data while the user interacts with an application. The challenge: in a horizontally scaled environment, any server may receive any request.

Sticky Sessions (ELB Feature)

How It Works

ELB routes a user's requests to the same server using client-side cookies. Cost-effective and speeds up retrieval.

Disadvantages

If the instance fails, all its sessions are lost. Creates uneven load distribution. Limits scalability and increases latency.

External Session Storage (Recommended)

Distributed Cache Cluster

Sessions stored in ElastiCache (Redis or Memcached). Any server in the Auto Scaling group can retrieve any user's session. Fully fault-tolerant — no sessions lost if a server fails.

Amazon DynamoDB

Sessions persisted in a DynamoDB table. Each server queries DynamoDB for the session on each request. Millisecond-scale latency. Durable and highly available by default.

Database Caching — Amazon ElastiCache

ElastiCache provides web applications with an in-memory data store in the cloud. It works as a side cache — supplements your primary database by removing unnecessary pressure, typically in the form of frequently accessed read data.

FeatureMemcachedRedis
Sub-millisecond latency✅ Yes✅ Yes
Scale horizontally for writes & storage✅ Yes (up to 20 nodes)❌ No (up to 250 nodes, vertical)
Multi-threaded performance✅ Yes❌ No

Rule of Thumb: Choose Memcached when you need simplicity, horizontal write scaling, and multi-threading. Choose Redis when you need persistence, advanced data structures, pub/sub, Multi-AZ failover, or leaderboards/sorted sets.

Caching Strategies

Lazy Loading

Only load data into cache when requested. On cache miss: fetch from DB, write to cache. Avoids filling cache with unused data. Disadvantage: first request is slow (cache miss penalty). Stale data possible if not combined with TTL.

Write-Through

Write to cache and DB simultaneously on every update. Cache is never stale. Disadvantage: write penalty on every update. Cache may fill with data that is never read. Best combined with TTL to expire unused keys.

DynamoDB Accelerator (DAX)

DAX is a fully managed, highly available, in-memory cache for DynamoDB. It delivers performance improvements of up to 10x — from milliseconds to microseconds — for read-heavy and bursty workloads. Fully integrated with DynamoDB, flexible, secure, and scalable.

Exam Tip — Caching Services Quick Reference

CloudFront: CDN edge caching for static/dynamic content + DDoS protection. ElastiCache Redis: in-memory side cache for RDS, persistence, pub/sub, Multi-AZ. ElastiCache Memcached: simple, horizontally scalable in-memory cache. DAX: in-memory cache specifically for DynamoDB (microsecond response). DynamoDB: session state persistence with millisecond response.

Planning for Disaster Recovery

Lec9_CSVC_Planning_for_Disaster_Recovery_VE.pptx — RPO, RTO & DR Patterns

The DR Mindset

"Everything fails, all the time." — Werner Vogels (AWS CTO). Disasters range from small-scale events (single instance failure) to large-scale (AZ outage) to colossal (regional disaster). You must design and plan for all of them intentionally.

AWS Well-Architected Framework Principles: Operational Excellence — "Anticipate failure" and "Refine operational procedures frequently." Reliability — "Test recovery procedures" and "Automatically recover from failure."

RPO & RTO — The Two Key Metrics

RPO — Recovery Point Objective
Maximum Acceptable Data Loss
The maximum amount of data (measured in time) that can be lost during a disaster. It answers: "How often must data be backed up?" Example: RPO of 8 hours → back up at least every 8 hours.
RTO — Recovery Time Objective
Maximum Acceptable Downtime
The maximum amount of time a business process can remain out of commission after disaster strikes. It answers: "How quickly must applications and data be recovered?" Example: RTO of 1 hour → be back online within 1 hour of the disaster.

Storage Backup Building Blocks

S3 Cross-Region Replication

Most S3 storage classes replicate data across AZs within a single Region. Configure CRR for higher-level data security across Regions. Automatically and asynchronously replicates newly created objects. Also reduces latency for geographically distributed users.

EBS Volume Snapshots

Point-in-time snapshots of EBS volumes stored in S3. Incremental (only changed blocks). Can be copied across Regions. Restore data to a new EBS volume. Use Amazon Data Lifecycle Manager to automate creation, retention, and deletion. Cannot snapshot EC2 instance storage.

File System Replication

Replicate EFS or FSx for Windows File Server file systems across Regions using AWS DataSync. Also replicate on-premises file systems to the cloud. Optional AWS Direct Connect for fast, consistent transfers.

Compute & Networking for DR

Custom AMIs & Auto Scaling

Obtain and boot new server instances within minutes using pre-built AMIs or container images. Prefer automated rebuild from configuration/code repos over instance-level backups. Use cross-region AMI copies for regional DR.

Route 53 Failover

Configure health checks on primary endpoints. Route 53 automatically switches traffic to a secondary endpoint when the primary fails. Essential for multi-region DR architectures.

AWS Direct Connect

Fast, consistent replication and backups of large on-premises environments to the cloud. Provides dedicated private connection — avoiding internet variability.

Database DR Features

Amazon RDS

Take snapshot data and save it in a separate Region. Combine read replicas with Multi-AZ deployments for a resilient DR strategy. Retain automated backups (up to 35 days). Multi-AZ provides synchronous standby replica for automatic failover.

Amazon DynamoDB

Back up entire tables in seconds. Point-in-time recovery (PITR) continuously backs up tables for up to 35 days — restore to any second within the window. Global Tables: multi-region, multi-master database for fast local performance and global availability.

Automation Services for DR

AWS CloudFormation

Use templates to quickly deploy collections of resources in a new Region or VPC in minutes. Duplicate production environments on demand. Fully repeatable, version-controlled infrastructure deployment.

AWS Elastic Beanstalk

Quickly redeploy your entire application stack in only a few clicks. Handles infrastructure automatically — focus only on your code.

AWS OpsWorks

Automatic host replacement. Combine with CloudFormation in the recovery phase — provision a new stack that supports the defined RTO using Chef or Puppet automation.

DR Best Practices

Start Simple

Begin with Backup & Restore and move to more complex patterns as your business needs dictate. Don't over-engineer from the start.

Check Licensing

Software licensing may impose restrictions on running instances in a DR region. Verify before your DR event — not during it.

Practice Game Day

Regularly simulate disaster scenarios and practice recovery procedures. You don't want to learn your DR plan doesn't work during an actual disaster.

Exam Tip — DR Pattern Decision Tree

Lowest cost, can tolerate hours of downtime? → Backup & Restore (S3 + Storage Gateway). Need faster recovery for core services? → Pilot Light (replicate data, stopped servers). Business-critical, need minutes RTO? → Warm Standby (scaled-down always running). Mission-critical, automatic failover, money no object? → Multi-Site (full active-active). Storage Gateway for on-premises-to-cloud backup in any pattern.
